Flow Diagram

The following diagram shows the process used to translate actions into code.

The following four steps describe the action translation process depicted in the figure above.

  1. The action translation process is initiated with an invocation of the .al_xlate rule command. The command is described in Invoking The Action Translation Process. The OAL statements contained within the Action_Semantics attribute of the action location are loaded into the action translation process.
  2. The action translation process parses the OAL and breaks it into a sequence of .invoke translation rules used to build up the action piece-by-piece. Each of the invoke rules specify a call to a fragment translation function. The result of this step is the creation of an intermediate rule file that contains calls to fragment translation functions used to build up the whole body for the action.
  3. The intermediate rule file is then interpreted. Each of the fragment translation functions are invoked and interpreted against the function definitions provided by the model compiler. Each function returns a small piece of translated OAL in a fragment. These fragments are combined, explicitly through calls to other fragment translation functions and parameter passing. At the end of this step the text accumulated in the fragment buffer is placed in the calling translation rule's buffer.
  4. The result of the .al_xlate command is a fragment whose body attribute contains the translated code for the specified action. The fragment is placed into the calling rule's literal text buffer upon return from the call to .al_xlate . The contents of the calling rule's buffer can then be saved to a file using the .emit command.
