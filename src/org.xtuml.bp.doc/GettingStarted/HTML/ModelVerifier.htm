Using BridgePoint Verifier to Debug the Sample Models

Overview

This section provides a brief introduction to the BridgePoint Verifier. It is meant to be a step-by-step guide enabling you to familiarize yourself with the tool, its look and feel, and how to get around. By the end of this section you should be able to open, execute, and debug models you have created.� BridgePoint Verifier allows you to interactively debug a model and its action language, without first translating the model into a target language. The BridgePoint Verifier interface is the xtUML Debugging perspective inside Eclipse.

Using BridgePoint Verifier

Start BridgePoint Verifier

Use the xtUML Modeling perspective (Window > Open Perspective > xtUML Modeling) to Launch Verifier on the MicrowaveOven model.���

After the selection has been made, a Verifier configuration dialog will come up.� Enter a configuration name, and check the box next to the MicrowaveOven model.� If you would like to see all transitions in the Console, check the Log model execution activity box.

Then select Debug.� The xtUML Debugging perspective will automatically be opened and the Verifier execution will begin.� The views in this perspective may be rearranged by drag-and-drop to suit your tastes.

Notice, in the Session Explorer view that none of the classes have any session information at this point.

Execute an Initialization Function

In order to execute the model we must first define some instances of the classes in the model.� To do this we will execute the DefineOven function.

Find the function in the Session Explorer view.� Right-click and select Execute.�

Notice the changes in the Session Explorer, instances are now present for all classes that had instances created by the OAL in the DefineOven function:

Browse Instance Data

From the Session Explorer, open the Magnetron Tube class. There is one instance in the list with an id value of something like 2.

Open up the Magnetron Tube instance to see the values of its attributes. The current state of this instance is shown to the right of the state icon �as Output Power Stable and OFF.

Browse an Instance State Machine

Use Model Explorer to open the Magnetron Tube class� Instance State Machine.� Select the Show entire canvas toolbar icon. �

Use the Spotlight View

In Session Explorer, right-click the Magnetron Tube instace and select Put in Spotlight.

The chosen instance will now appear in the Spotlight view and the current state of the Magnetron Tube instance (Output Power Stable and OFF) is shown highlighted in the Magnetron Tube: Instance State Machine diagram.

Note: Only instances in Spotlight affect the current state highlighing behavior of the state machine diagram. Classes with state machines may also be added to Spotlight.

Set a Breakpoint

Use Model Explorer to open the TestSequence1 function.� Right-click in the margin to the left of the action language.� Use the context menu to toggle a breakpoint on.

The breakpoint you just set is shown in the Breakpoints view and can be enabled and disabled from there (as well as directly in the OAL editor by right clicking). Select Breakpoint Properties... from either place. There is a Hit count you can set in this dialog.

Once you have set the breakpoint, right-click on the function named TestSequence1 in Model Explorer and select Execute.

Execution suspends when the action language in TestSequence1 is executed and the hit count is reached. The Variables view shows the current values of any action language temporary variables in the OAL. The Debug view shows any events that were in the event queue when execution suspended.

State Machine Animation

Bring the Magnetron Tube: Instance State Machine diagram back to the foreground and select the Resume button �in the Debug view.� The highlighted state in the diagram and in the Spotlight will change as the model executes.�

Execution Control

As with code debuggers, Suspend and Terminate are available for execution control.� Stepping is also supported to execute action language line by line.� The Variables view will be updated as stepping proceeds.

Other Features

�         Breakpoints can be set on events. You can set Queued, Dequeued, Ignored and Can't Happen breakpoints.� Locate the event in Model Explorer.� Right-click and select Set Breakpoint in the context menu. Next, in the Breakpoints view, right-click on the desired event breakpoint and select Breakpoint Properties�

�         Breakpoints can also be set on states. You can set Transition Into and Transition Out of breakpoints.� Locate the state in Model Explorer or Session Explorer.� Right-click and select Set Breakpoint in the context menu. Next, in the Breakpoints view, right-click on the desired state breakpoint and select Breakpoint Properties�

�         Multiple instances can be shown in the Spotlight view allowing visual tracking of the current states of multiple instances and classes (classes that have state machines can also be added to the Spotlight view).

�         Step Over and Step Into work for events shown in the Debug view event queue, too. If an event is delayed, Step Over and Step Into will dequeue the event immediately without waiting for the time to expire.

�         The model may be modified while Verifier is running and the changed behavior will be seen immediately.

�         Control can be transferred from a bridge call to a user supplied Java class.� The class must reside in the package lib in the root of the src/ folder. The class name must match the key letters of the External Entity and the method must match in name and parameter types those declared for the bridge.
